name: developer-compliance-gate
description: Use when a developer must translate compliance evidence and gaps into concrete implementation tasks, acceptance criteria, and release gates for a government digital assistant. Trigger phrases include "developer tasks", "acceptance criteria", "release gate", and "evidence pack".
---

## When to use this Skill

Use when intake or evidence-pack findings exist and the team needs engineering work items, acceptance criteria, release gates, and a Definition of Done.

If no evidence pack is pasted, ask once. If the user says "vertaal dit evidence pack" without content, use gaps from the demo evidence pack (missing source freshness checks, partial audit logging, partial human escalation wiring) and label as demo assumptions.

## Inputs to collect

- Intake assessment, evidence pack, risk register, and readiness status
- RAG architecture, source metadata model, prompt handling, logging design, model versioning
- Target release gate: pilot or production
- Owners, repositories, environments, synthetic test datasets

## What to do

1. Follow these steps after loading this skill.
2. Map each compliance gap to at least one developer task with acceptance criteria.
3. Include tasks for: source citation and RAG metadata; source allowlists; prompt-injection handling; refusal behavior; escalation behavior; personal-data redaction; audit logging; model/provider version recording; retrieval trace recording; monitoring dashboards; alerting; incident runbook; rollback path; environment separation (experiment / pilot / production); synthetic test data only.
4. Define logging fields: request ID, timestamp, assistant version, model/provider version, source IDs, retrieval parameters, refusal/escalation flags, reviewer-safe redaction.
5. Define audit trail requirements without storing raw personal data by default.
6. Define security checks: secrets handling, prompt injection, source allowlists, dependency scanning, access control.
7. Define monitoring metrics: answer quality, refusal rate, escalation rate, source freshness, incident count, latency, cost, user feedback.
8. Choose exactly one release status: `passes developer gate`, `passes with conditions`, or `does not pass developer gate`.

Unresolved privacy, security, or human-oversight gaps must **not** result in `passes developer gate`.

## Consistency contract

Consume evidence-pack rows as the source of truth. Keep the original control name in every issue so product, privacy, security, and development can trace work back to the evidence pack.

Preserve incoming intake `Handoff ID` and evidence `Evidence ID` values in issue titles or metadata. If the evidence pack has no IDs, create `DEV-GAP-<number>` IDs and state that they were generated by this Skill.

For each issue, write acceptance criteria that are observable in at least one of:

- a pull request diff
- a synthetic test
- an audit-log sample with redacted data
- a dashboard or alert
- a runbook, rollback plan, or deployment setting

Prefer platform-neutral tasks. If the user asks for a kagent or Kubernetes demo, add a short optional `kagent showcase` note that maps the task to agent logs, skill loading, or GitOps deployment evidence, but keep the Skill itself usable outside kagent.

## Output format

```markdown
# Developer compliance gate

## Release status
<passes developer gate | passes with conditions | does not pass developer gate>

## Implementation checklist
- [ ] Source citation + RAG metadata on every answer
- [ ] Source allowlist enforced at retrieval
- [ ] Prompt-injection detection + safe refusal
- [ ] Escalation to human for legal/case-specific questions
- [ ] PII redaction in logs and exports
- [ ] Audit log: request ID, timestamps, model version, source IDs, refusal flags
- [ ] Retrieval trace stored (no raw citizen PII by default)
- [ ] Monitoring dashboards + alerts (refusal, escalation, freshness, latency)
- [ ] Incident runbook + rollback to non-AI fallback
- [ ] Environment separation experiment/pilot/production
- [ ] Synthetic test data only in CI/staging

## Suggested GitHub issues

### Issue: Implement audit logging for assistant requests

**Traceability ID:** EVID-AUDIT-1

**Why**
Evidence pack marks logging as partially satisfied; pilot requires traceable requests without storing raw PII.

**Implementation tasks**
- [ ] Add structured log schema (request_id, assistant_version, model_version, source_ids, refusal_flag)
- [ ] Redact PII fields before write
- [ ] Wire log shipping to central store

**Acceptance criteria**
- [ ] Every assistant response produces one audit record
- [ ] No raw citizen PII in default log stream
- [ ] Retrieval parameters and source IDs present for sample traffic

### Issue: Enforce RAG source allowlist and freshness

**Traceability ID:** EVID-RAG-1

**Why**
Missing source governance blocks pilot sign-off.

**Implementation tasks**
- [ ] Config-driven allowlist per environment
- [ ] Freshness metadata on indexed documents
- [ ] Alert when stale source used in answer

**Acceptance criteria**
- [ ] Retrieval rejects non-allowlisted sources
- [ ] Answers cite source ID + freshness timestamp
- [ ] Stale-source usage visible in metrics

### Issue: Human escalation path for high-risk questions

**Traceability ID:** EVID-OVERSIGHT-1

**Why**
Human oversight plan requires escalation for legal and case-specific queries.

**Implementation tasks**
- [ ] Classifier or rule-based escalation triggers
- [ ] Handoff to case worker queue with conversation context (redacted)
- [ ] User-facing transparency message on escalation

**Acceptance criteria**
- [ ] Escalation fires on configured trigger set in synthetic tests
- [ ] Escalation rate metric exported
- [ ] No automated decision on escalated threads

Passing this developer gate means the requested implementation evidence is present or tracked. It is not legal approval.
```

Provide **3–5** GitHub issues for the demo, prioritizing gaps marked `missing` or `partially satisfied`.

## What to avoid

- Do not pass the gate while privacy, security, or human-oversight gaps remain unresolved.
- Do not ask developers to use real citizen data in tests.
- Do not require kagent, Kubernetes, or a specific platform for Skill execution.
- Do not include secrets or raw personal data in issue bodies or logs.
